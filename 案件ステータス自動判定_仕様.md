# 案件ステータス自動判定 仕様書（会話履歴 → data.json 反映）

最終更新: 2026-06-02 ／ 対象: `anken-summary` ・ `AR2026チェック` などのスキル層

---

## 0. 位置づけ（重要）

- 本書は **スキル（Cowork / Claude）が従う判定ルールの仕様**です。
- `genba-navi` のブラウザアプリ（静的HTML）は **Cドライブ/Gドライブの共有フォルダや会話履歴を読めない**（GitHub Pages 上の `data.json` しか取得できない）ため、本処理は **スキル側で実行**します。
- 既に実装済みの「見積/請求 保存 → ステータス前進更新」（`mitsumori_seikyu_preview.html`）と **同じステータス語彙・前進のみルール**で一貫させます。

---

## 1. 目的

各案件に紐づく **会話履歴・案件サマリー** を読み取り、案件の進捗イベントを検出して
`data.json` の `案件[].ステータス` を **前進方向のみ** 自動更新する。
人手の更新漏れを減らしつつ、**後退・誤上書き・看板からの脱落を防ぐ**。

---

## 2. 入力（※「要確認」は実運用に合わせて確定してください）

| 入力 | 内容 | 状態 |
|---|---|---|
| 案件サマリーHTML | `AR-2026/<案件>/` 配下。`anken-summary` スキルが会話から生成するスナップショット | 確認済み（手順書/要件に記載） |
| 共有フォルダA | （プロジェクト共有フォルダ 1つ目） | **要確認: フォルダ名/パス** |
| 共有フォルダB | （プロジェクト共有フォルダ 2つ目） | **要確認: フォルダ名/パス** |
| 共有フォルダC | （プロジェクト共有フォルダ 3つ目） | **要確認: フォルダ名/パス** |
| 履歴ファイル形式 | 上記フォルダ内の履歴ファイルの形式（HTML/MD/TXT/チャットログ等）と、案件キーの埋め込み方 | **要確認** |

> 案件の特定キー: `data.json` の `案件キー`（例: `ネクサス-大丸引戸修理-26`）または `旧案件番号`（例: `AR-26-011`）。
> 履歴ファイル側にこのいずれかが含まれることを前提とする（含まれない場合は現場名/顧客名で曖昧マッチ→候補提示）。

---

## 3. ステータス語彙（dashboard の KANBAN_COLS と完全一致）

パイプライン順（前進=右方向）:

```
相談 → 見積準備 → 見積提出済み → 施工予定 → 施工中 → 請求済み → 入金済み
（クローズゾーン）完了・請求書作成待ち → 完了(請求書作成済み) → 完了 → 保守管理(継続) → 中止 → 消滅
```

- ここに **無い語を入れない**こと（看板のグルーピングから外れて案件が表示されなくなる）。
- `中止` `消滅` は **明示の合図がある時だけ**設定（自動前進では触らない）。

---

## 4. 判定ルール（履歴の合図 → 目標ステータス）

検出した最も進んだイベントを「目標ステータス」とする。複数該当時は **パイプライン上で最も先のもの** を採用。

| 履歴に現れる合図（例・正規化後） | 目標ステータス |
|---|---|
| 「相談」「問い合わせ」「現調依頼」「下見」「調査中」「現地調査」 | 相談 |
| 「図面作成」「平面図作成」「作図」「実測図」「製図」「CAD」「図面を描いた」「図面制作」 | 見積準備 |
| 「見積作成」「積算中」「拾い出し」 | 見積準備 |
| 「見積提出」「見積送信」「見積書を送った」「御見積」 | 見積提出済み |
| 「受注」「契約」「発注書受領」「着工日確定」「施工予定」 | 施工予定 |
| 「着工」「施工中」「工事開始」 | 施工中 |
| 「完工」「施工完了」「工事完了」「引き渡し」 | 完了・請求書作成待ち |
| 「請求書作成」「請求送信」「請求済み」 | 請求済み |
| 「入金確認」「入金済み」「振込確認」 | 入金済み |
| 「保守」「定期点検」「継続管理」 | 保守管理(継続) |
| 「中止」「キャンセル」「失注」「見送り（確定）」 | 中止（※明示時のみ） |

> 注: 音声入力（Aqua Voice）由来の誤変換に備え、表記ゆれ・同義語を許容。
> 注: コワーク/チャットが案件作成時に「相談」または「調査中」のどちらでも発動しうる。
> 「調査中」は「相談」とほぼ同義として扱い、看板では両方とも**営業ゾーン**に並列表示する
> （データ上のステータス文字列は統合せず、看板側で両方を営業ゾーンの列として受け入れる）。
> 「請求」は **見積/請求の保存イベント**（実装済み）とも整合（請求保存→請求済み）。
> 注（2026-06-08 追加）: **図面作成（平面図・実測図・作図）を進捗の起点として扱う**。
> 自社案件では「図面 → 見積」の順で進むケースが多く（むしろ図面先行が主流）、図面作成は
> 見積の前提作業（材料割付・数量確定の土台）にあたるため **「見積準備」へ前進** させる。
> 看板の語彙を増やさない方針（§3）に従い、図面専用ステータスは設けず「見積準備」に集約する。
> ※本対応表は運用に合わせて随時更新していく前提（合図の追加・調整は本書を都度改訂）。

## 5. 更新ロジック（前進のみ・安全側）

実装済みの保存機能と同一ルール:

```
cur  = 現在のステータス
tgt  = 判定した目標ステータス
ci   = PIPELINE.indexOf(cur)
ti   = PIPELINE.indexOf(tgt)

更新する条件:
  - cur が「中止」でない
  - tgt がパイプラインに存在する (ti !== -1)
  - かつ ( ci === -1（現在が未知/未設定）または ti > ci（目標が現在より先） )

→ 上記を満たす時のみ 案件[].ステータス = tgt
→ それ以外（後退・同値・完了系より手前）は変更しない
```

- **後退させない**（例: `入金済み` の案件を履歴の古い「見積送信」で `見積提出済み` に戻さない）。
- `完了` `完了(請求書作成済み)` `保守管理(継続)` `中止` `消滅` などクローズ済みは原則据え置き。
- `中止` への遷移は **明示の合図がある時のみ**（自動前進の対象外）。

---

## 6. 反映（書き込み）方法

スキルが `data.json` を更新する経路は次のいずれか。**衝突対策（task 4 既出）と一貫**させること。

- **経路A（推奨・ローカル起点）**: Cドライブの `data.json` を直接編集 → 安全版 `auto_push.bat`（pull衝突時は中断）で GitHub へ反映。
- **経路B（GitHub API）**: ダッシュボード/見積請求の保存と同じ Contents API。最新SHA取得→更新→PUT、競合(409/422)時は再取得して再適用。

いずれも:
- **1回の更新で対象案件のみ**を最小編集（差分を小さく→マージ衝突を避ける）。
- 変更時は証跡を残す（後述）。

---

## 7. 証跡・冪等性

- 更新した案件には、根拠を残す。例: `案件[].備考` に追記、または `案件[].ステータス更新履歴`（任意の配列）に
  `{日時, 旧→新, 根拠ファイル, 検出合図}` を1件追加。
- **冪等**: 同じ履歴を再処理しても、前進条件を満たさなければ無変更（二重更新しない）。
- 1案件1回の更新に留め、ログ（処理対象・結果・スキップ理由）を出力。

---

## 8. 例

- 現在 `施工中` の案件サマリーに「請求書送信済み」→ 目標 `請求済み`（ti>ci）→ **更新**。
- 現在 `請求済み` に履歴の古い「見積提出」→ 目標 `見積提出済み`（ti<ci）→ **据え置き**。
- 現在 `完了` に「着工」→ 据え置き（クローズ済み）。
- 履歴に「失注・中止確定」→ `中止`（明示合図）。
