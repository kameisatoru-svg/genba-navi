# 見積単価マスター反映スキル（mitsumori-tanka-hanei）

## 何のためのスキルか

見積は「雛形の参考単価 → かめさんが現場感覚で修正 → PDF化して客先へ」という流れで作る。
**この“修正後の単価”こそが実際に通用した単価**なのに、これまでは案件フォルダの中に
埋もれたままで、次の見積に活きていなかった。このスキルはその単価をマスターへ環流させる。

- 入力：`AR-2026\<顧客>\<案件キー>\書類\見_*.html`（かめさんが修正・PDF化した見積の元HTML）
- 出力①：`単価マスター_アートレイズ.html`（工種別単価マスター ver.2）の更新
- 出力②：`jisseki_tanka.json`（実績単価の正本。案件・発行日・出典つきで全件蓄積）

`sanko_tanka.json`（過去工事参考単価／みつも郎PDF由来）とは別レイヤー。
あちらは**過去の集計**、こちらは**かめさんが今出している単価**。混ぜない。

---

## 0. データの流れ

```
見_*.html の <script id="artrays-doc-state"> （明細JSON）
        │  extract_tanka.py（読み取り専用）
        ▼
teian_tanka.json  ← 反映候補（判定つき）＋ Markdown表をチャットへ
        │  かめさんが採否を指示
        ▼  apply_tanka.py
単価マスター_アートレイズ.html ＋ jisseki_tanka.json
```

**書き込みは apply_tanka.py だけ**。extract 側は絶対に書かない（誤反映の事故を構造で防ぐ）。

---

## 1. 判定の4区分

| 判定 | 意味 | 既定の扱い |
|---|---|---|
| **レンジ外** | マスターに同名・同単位の行があり、実績がそのレンジ外 | レンジを実績まで広げる提案 |
| **未登録** | マスターに該当行が無い | 工種を推定して新規行を追加する提案 |
| **要確認** | マスターが「実費」等、または実績とマスターが3倍以上乖離 | **自動反映しない**（別項目の誤マッチを疑う） |
| **レンジ内** | 実績がレンジに収まっている | 変更不要（実績記録のみ） |
| **一式記録** | 単位が「式」＝案件固有の一括計上 | **マスター対象外**（記録のみ／`--include-isshiki` で例外） |

### 突合の鉄則

- **単位が違う行は別物として扱う（マッチさせない）**。
  実害例：`壁開口(箇所 8,000〜15,000円)` に `壁開口補強 W-65(m 1,350円)` が誤マッチし、
  レンジが `1,350〜20,000円` という使い物にならない幅に壊れた。単位一致は必須条件。
- 名称は NFKC 正規化＋空白・記号除去で突合し、類似度 0.82 以上を同一とみなす（`--cutoff` で調整）。
- **3倍以上の乖離は「要確認」に落として自動反映しない**。桁違いは大抵が誤マッチ。

### 20%アップとの関係（重要）

見積HTMLに入っている単価は、**すでに値上げ分（過去単価×1.2）を織り込んで客先に出した数字**。
マスターへはその数字を**そのまま**入れる。マスターを見て次の見積を作るときに、
さらに1.2を掛けない（[見積単価は20%アップが既定] は"過去単価ベース"に対する運用）。

## 2. 手順

### Step 1｜候補を出す（読み取りのみ）

```bash
cd "C:\Users\user\artrays\claude ai\genba-navi"
python ".claude/skills/mitsumori-tanka-hanei/scripts/extract_tanka.py"
```

- 既定は**差分モード**：`jisseki_tanka.json` の `処理済み`（sha1）に無い／変わった見積だけを見る。
- `--all` 全件洗い直し、`--since 2026-07-01` 日付以降、`--files a.html b.html` 個別指定。
- `--include-seikyu` を付けると `請_*.html` も対象（既定は見積のみ）。
- 標準出力の Markdown 表をそのままチャットへ貼り、`teian_tanka.json` に候補を保存する。

### Step 2｜かめさんに採否を取る

**マスターの書き換えは必ず承認を取ってから**。表を提示し、次のどれかを確認する。

- 「レンジ外は全部反映」→ `--all-rangeout`
- 「未登録も全部追加」→ `--all-new`
- 個別に選ぶ → `--pick "ソフト巾木貼|m,床長尺シート貼|㎡"`（`名称|単位` をカンマ区切り）
- 件数が多いときは `名称|単位` を1行ずつ書いたテキストを作って `--from-file`

> 未登録が数十件出るのは正常。マスターは業界一般値、かめさんの見積は現場語なので語彙が
> ほぼ重ならない。**一気に全部入れず、頻出・再利用しそうなものから選ぶ**のを勧めること。
> 工種が `（要手動分類）` の候補は、工種を確定させてからでないと追加できない（見送りになる）。

### Step 3｜反映する

```bash
python ".claude/skills/mitsumori-tanka-hanei/scripts/apply_tanka.py" --dry-run --all-rangeout
python ".claude/skills/mitsumori-tanka-hanei/scripts/apply_tanka.py" --all-rangeout --pick "..."
```

- **必ず `--dry-run` を先に通し、変更内容を読んでから本実行**する。
- `--record-only` はマスターに触れず `jisseki_tanka.json` への記録だけ行う。
- 実績記録は採否に関わらず**全候補**が入る（マスターに入れなくても履歴は残す）。

### Step 4｜確認

- auto-push が数分で GitHub Pages へ配信する。**git 操作は不要**。
- 差分は `git diff -- 単価マスター_アートレイズ.html` で確認できる。

## 3. 安全装置（実装済み・崩さないこと）

| 装置 | 内容 |
|---|---|
| バックアップ | 書き込み前に `_アーカイブ\単価マスター_バックアップ\*.bak` へ複製 |
| アトミック書き込み | 一時ファイル→内容照合→`os.replace`（data.json 破損の教訓） |
| 行数検証 | 書き込み後に再パースし、`元の行数＋追加数` と一致しなければ異常終了しバックアップ位置を表示 |
| 冪等性 | 同名・同単位が既にあれば追加しない。差分モードは sha1 で処理済み見積を飛ばす |
| 読み書き分離 | 抽出スクリプトは一切書き込まない |
| 削除しない | ファイルは消さない（[最優先原則0] `rm` 禁止・退避は `_破棄\`） |

## 4. トラブル対応

- **`teian_tanka.json がありません`** → Step 1 を先に実行する。
- **候補が0件** → 差分モードで全部処理済み。`--all` で洗い直す。
- **`マスター行が見つからない`** → マスターを手で編集して名称が変わった可能性。Step 1 からやり直す。
- **`工種が未確定（要手動分類）`** → `teian_tanka.json` の該当候補の `工種` を正しい工種名
  （例 `06｜内装仕上工事`）に手で書き換えてから apply する。
  同じ誤分類が繰り返されるなら `extract_tanka.py` の `SECTION_RULES` にキーワードを足す。
- **状態JSONの無い古い見積HTML** → 明細表を素で読むフォールバックが働く。拾えていなければ
  その案件は手入力で `--pick` する（現状 26件中2件が該当）。

## 5. スクリプトの置き場所（触るときの注意）

**正本は `genba-navi\.claude\skills\mitsumori-tanka-hanei\scripts\` の1箇所だけ**。
Cowork のプラグイン配下（AppData）にはこの SKILL.md しか置かない。
理由：プラグイン配下はパスが長く、`scripts\extract_tanka.py` まで並べると **Windows の
260文字上限を超えて Python がファイルを開けない**（実測261文字で `Errno 2` になった）。
スキルを直す場合も、コードは常にリポジトリ側を編集し、SKILL.md だけ両方に反映する。
