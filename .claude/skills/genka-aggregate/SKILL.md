# 原価集計反映スキル（genka-aggregate）

原価明細（RC-26-xxx）を集計し、各案件の4費目原価（労務費・材料費・外注費・経費）を
`genka.json`（明細の正本）と `data.json`（案件台帳のミラー）へ反映する手順。

初版のきっかけ：AMEX 2026-06分（RC-26-091〜135／45件）を貼付用シートから直接読み取って反映した作業。
**経理の金額を扱うため、検算・バックアップ・厳密検証を必須**とする。

---

## 0. データモデル（正本＝genka.json）

`ai_assist.html` 記載の構造が正本仕様。

```
genka.json = { 最終集計日,
  案件: { "<案件キー>": {
     旧案件番号,                       // 例 "AR-26-004"（共通経費は ""）
     分類: {労務費,材料費,外注費,経費}, // 4費目集計
     合計, 件数, 最終集計日,
     明細: [ {id, 日付, 店舗, 品目, 金額, 勘定科目, 原価区分, 支払方法, ソース, 備考} ... ]
  } } }
```

- `data.json` 側はミラー：`案件[].原価 = {労務費,材料費,外注費,経費,合計,件数,最終集計日}`。
- **突合**：明細の `案件番号`(AR-26-xxx) → genka/data の `旧案件番号` で案件キーを解決。
- **共通経費**：genka では案件キー `"共通"`（旧案件番号は空文字）。**data.json には「共通」案件は無い**ため、
  共通分は **genka.json のみ更新**し data.json へは反映しない。

### 費目マッピング（重要・必ず検算）

| 勘定科目 | 振り分け先 |
|---|---|
| 材料費 | 材料費 |
| 外注費 | 外注費 |
| 労務費 | 労務費 |
| 上記以外すべて（消耗品費・旅費交通費・交際費・会議費・車両費・諸会費・事務消耗品費・管理諸費・雑費 等） | 経費 |

> 着手時に**既存の全案件で「明細→分類 再計算 == 保存値」を検算**し、0件不一致を確認してから新規行を足す。
> （マッピングが将来変わっても、この検算で気づける。）

|---|---|---|
| 本体台帳 | スプレッドシート「原価管理」 | `1f_SXMlN07czsI7YsPMvv8bQhsWpU5Aiyg_HtM9L0LdM`（最終行のRC番号＝集計済みの上限） |
| 貼付用（月次） | 「原価管理_AMEX<YYYY-MM>_貼付用」 | 例 2026-06 = `1hOVZSQb0lf1N4y0f7ToPdTCx-fw_V9dCXd-iZ_SpJoA` |

  natural language の `read_file_content` は確認用。**金額・勘定科目・案件番号は転記ミス厳禁**なのでCSV経由が安全。

## 2. 反映する行の決め方

1. genka.json の既存明細IDを全収集し、**最大RC番号**を把握。
2. 「本体台帳の最終行RC」と「貼付用シートのRC範囲」を確認。
3. **新規行 = genka.json に未登録のID全部**（＝本体台帳に貼られたが未集計の分も含める）。
   - 例（初版）：genka最大=090、本体=093、貼付用=094〜135 → 新規は **091〜135**
     （091〜093は本体に貼られて未集計だった分）。
4. **重複ガード**：新規候補のうち既存IDに含まれるものは除外（再集計しない）。

---

## 3. 集計と書き込み（安全手順）

ツール：`python3`（標準ライブラリのみ）。

1. **検算**（§0の費目マッピングを既存全案件で照合、不一致0を確認）。
2. **CSV取得**：貼付用を `download_file_content(text/csv)` → base64デコード。
   行数・全列・金額が数値・RC範囲を検証。本体のみの未集計行（例 091-093）は本体台帳の値を使う。
3. **バックアップ**：`genka.json.bak_<YYYYMMDD_HHMMSS>` / `data.json.bak_<...>` を作成。
4. **genka.json 更新**：各新規行を該当案件の `明細` 末尾に**昇順を保って追記**、
   `分類[bucket]+=金額` / `合計+=金額` / `件数+=1`、案件と最上位の `最終集計日` を当日に更新。
   追記後、案件ごとに「分類再計算==保存」「Σ分類==合計」「len(明細)==件数」を assert。
5. **data.json 更新**：旧案件番号で対象案件を引き、`原価` を genka の分類/合計/件数＋`最終集計日`でミラー。
   `最終更新` を当日時刻（JST `YYYY-MM-DD HH:MM`）に更新。**共通はスキップ**。
6. **書き込み書式（差分最小・破損防止）**：
   - `genka.json`：`json.dumps(..., ensure_ascii=False, indent=1)`（**1スペース**・末尾改行なし）
   - `data.json`：`json.dumps(..., ensure_ascii=False, indent=2)`（**2スペース**・末尾改行なし）
   - 事前に round-trip 一致（load→dump==元）を確認すると差分が値のみになる。
7. **厳密検証（バックアップと比較）**：
   - data.json：変更が `最終更新` と対象案件 `原価` **のみ**であること、案件数不変、他トップキー不変を assert。
   - genka.json：**未影響案件がすべて不変**、影響案件は**既存明細を順序保持**＋新規のみ追記、を assert。
8. **後始末**：作業用 `*.bak_*` は削除（git履歴が版管理）。`genka.json` と `data.json` のみ commit/push。

> data.json は過去に破損履歴が多い。**必ず json で load→dump し、書式は上記固定、書き込み後に再パース＋差分検証**する。

---

## 4. 報告と限界

- 反映後の各案件 `分類/合計/件数`、対象RC範囲、加算内訳を報告。
- **本体「原価管理」Sheetsへの行追記は自動化不可**（書き込みコネクタ無し）。台帳の記録として残すには、
  ユーザーが貼付用シートの新規行を本体最終行へ手貼りする（data.json反映は本スキルで完了済み）。
- 将来Sheets書き込みコネクタが入れば、§1の追記まで含めて全自動化する（その際は本書を更新）。
